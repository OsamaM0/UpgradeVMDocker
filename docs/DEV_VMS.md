# Developer VMs — SSH, PyCharm, and database access

Every `devs/<name>/` instance is a self-contained "VM": SSH straight into its
Odoo container, browse its own Odoo web UI over HTTPS, and connect to its own
isolated PostgreSQL database. No instance can see another instance's data,
addons, or containers.

## 1. Connect over SSH (the VM itself)
```bash
ssh -p <SSH_PORT> -i devs/<name>/keys/<name>_id_ed25519 developer@<VPS_IP>
```
- Password auth is disabled — key-only. The key is installed from the
  `DEV_SSH_PUBLIC_KEY` value in `devs/<name>/.env` (or the Dokploy Environment
  tab) every time the container starts.
- The `developer` user has passwordless `sudo` inside the container and is in
  the `odoo` group (can read/write addons and the filestore).
- Odoo's own Python 3 + all Odoo libraries are the system interpreter here —
  no separate venv to configure.

## 2. Open the Odoo web UI
Two options, pick either:
- **Public HTTPS** (if `DOMAIN` in `.env` has a DNS A record → this VPS):
  `https://<name>.example.com`
- **SSH tunnel** (works even without DNS/Traefik):
  ```bash
  ssh -p <SSH_PORT> -i devs/<name>/keys/<name>_id_ed25519 \
      -L 8069:127.0.0.1:<ODOO_PORT> developer@<VPS_IP>
  # then open http://localhost:8069
  ```

## 3. PyCharm

### Remote Python interpreter (runs inside the developer's own container)
Settings → **Python Interpreter → Add → SSH**
- Host: `<VPS_IP>` · Port: `<SSH_PORT>` · User: `developer`
- Auth: key file `devs/<name>/keys/<name>_id_ed25519`
- Interpreter path: `/usr/bin/python3`
- Sync folders: map your local addons checkout to `/mnt/extra-addons` on the remote.

This gives full code completion / debugging against the real Odoo 18
installation running in that developer's own container — not a shared one.

### Database (Professional)
Database tool window → **+ → Data Source → PostgreSQL**
- Host `localhost` · Port `<ODOO_local_forward>` (or use the **SSH/SSL** tab
  and let PyCharm open the tunnel via `<SSH_PORT>` automatically)
- User/password/db: from `devs/<name>/.env` (`POSTGRES_USER` / `_PASSWORD` / `_DB`)

## 4. VS Code — Remote-SSH
```
Ctrl+Shift+P → Remote-SSH: Connect to Host
ssh -p <SSH_PORT> -i devs/<name>/keys/<name>_id_ed25519 developer@<VPS_IP>
```
Opens a full remote workspace inside that developer's own container-VM.

## 5. Direct container access (from the VPS host)
```bash
docker exec -it <name>-app odoo shell -c /tmp/odoo.conf -d <name>
docker exec -it <name>-app bash
docker exec -it <name>-db psql -U <name> -d <name>
```

## 6. Importing a database into one developer's instance
`scripts/restore-db.sh` operates on the main stack's `.env`/containers. For a
`devs/<name>` instance, restore directly against that instance's own
containers (same dump formats: `.sql`, `.sql.gz`, custom `pg_dump`, Odoo `.zip`):
```bash
cd devs/<name>; set -a; . ./.env; set +a     # load this instance's credentials

# plain SQL
docker exec -e PGPASSWORD="$POSTGRES_PASSWORD" -i "${PROJECT_NAME}-db" \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" < /path/to/dump.sql

# custom pg_dump format
docker exec -e PGPASSWORD="$POSTGRES_PASSWORD" -i "${PROJECT_NAME}-db" \
  pg_restore -U "$POSTGRES_USER" -d "$POSTGRES_DB" --no-owner --role="$POSTGRES_USER" < /path/to/dump.dump

docker compose restart odoo
```

## 7. Rotating / adding SSH keys
Append more public keys (e.g. a second machine) to `DEV_SSH_PUBLIC_KEY` in
`devs/<name>/.env` (one key per line — it's a multi-line value), then:
```bash
cd devs/<name> && docker compose restart odoo
```
No key material lives in Git — keys are supplied only via `.env` /
the Dokploy Environment tab. The private key generated by
`scripts/new-dev-vm.sh` is git-ignored (`devs/*/keys/`) and must be handed to
the developer securely (and removed from the server) after generation.

## 8. Security notes
- SSH is the **only** public entry point per instance besides HTTPS. Odoo's
  raw port (`ODOO_PORT`) and PostgreSQL (`POSTGRES_PORT`) are bound to
  `127.0.0.1` — reachable only through that instance's own SSH tunnel.
- Each instance has its own SSH host keys (persisted in the `sshd-keys`
  volume), so instances never share host-key fingerprints.
- Firewall: open only `SSH_PORT` per instance (plus the shared 80/443 for
  Traefik) — see [SECURITY.md](SECURITY.md).
